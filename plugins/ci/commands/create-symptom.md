description: Interactively create (or update/delete) a Sippy Symptom that auto-labels a known CI failure pattern

## Name

ci:create-symptom

## Synopsis

```
/ci:create-symptom [description-of-failure]
```

## Description

The `ci:create-symptom` command creates a Sippy Symptom — a known-failure signature made of an artifact file pattern and a matcher that, when it matches a CI job run, automatically applies human-readable Labels (like `InfraFailure`) so future failures are recognized without re-debugging. The command walks through an interactive workflow: check for duplicate symptoms, verify (or create) the target labels, confirm the exact payload with the user, then create. It also handles updating or deleting an existing symptom when the user asks.

## Implementation

1. **Load the skills**: Use the `manage-symptoms` skill (primary), plus `list-symptoms` for the duplicate/label checks, `manage-labels` when a target label is missing, and `oc-auth` for the write token.

2. **Check for duplicates**: Search the existing catalog with terms from the user's failure description:
   ```bash
   python3 plugins/ci/skills/list-symptoms/list_symptoms.py --search "<keywords>" --format summary
   ```
   If an equivalent symptom exists, propose updating it instead of creating a new one.

3. **Verify the target labels exist**:
   ```bash
   python3 plugins/ci/skills/list-symptoms/list_symptoms.py --labels --format summary
   ```
   If a needed label is missing, create it first using the `manage-labels` skill (confirm title and explanation with the user).

4. **Obtain the auth token**: Follow the `oc-auth` token-acquisition steps in the `manage-symptoms` skill (requires `oc login` to the DPCR cluster, `https://api.cr.j7t7.p1.openshiftapps.com:6443`).

5. **Confirm the payload with the user**: Show the full payload — summary, matcher type (string|regex|none|cel), file pattern, match string, label IDs — and get explicit confirmation before sending.

6. **Create the symptom**:
   ```bash
   python3 plugins/ci/skills/manage-symptoms/manage_symptoms.py create \
     --token "$TOKEN" \
     --summary "<short unique summary>" \
     --matcher-type string \
     --file-pattern "build-log.txt" \
     --match-string "<text that identifies the failure>" \
     --label-ids <LabelID>
   ```
   The symptom `id` is generated by the server from the summary.

7. **Update/delete on request**: For updates, run `manage_symptoms.py update --id <id>` passing only the flags to change (the script merges with the existing symptom). For deletes, first show the symptom (`list_symptoms.py --id <id> --format summary`) and get explicit user confirmation, then run `manage_symptoms.py delete --id <id>`.

8. **Verify**: Fetch the result with `list_symptoms.py --id <new-id> --format summary` and optionally preview matches on a known-affected run with the `reevaluate-job-runs` skill using `--dry-run`. Suggest `reevaluate-job-runs` to apply the new symptom to past runs.

## Return Value

- **Format**: The created/updated symptom JSON from the API, followed by a verification summary
- **Key fields**: id (server-generated), summary, matcher_type, file_pattern, match_string, label_ids

## Examples

1. **Create a symptom from a failure description**:
   ```
   /ci:create-symptom AWS install fails with "api error AuthFailure" in build-log.txt
   ```

2. **Update an existing symptom**:
   ```
   /ci:create-symptom update AWSCouldNotValidateAccessCredentials to also match the new error text
   ```

## Arguments

- $1: Description of the failure (optional) — free-form text describing the failure pattern, the file it appears in, and/or the label to apply. If omitted, ask the user for the details interactively.

## Skills Used

- `manage-symptoms`: Creates/updates/deletes symptoms via the authenticated Sippy API
- `list-symptoms`: Duplicate check, label verification, and post-create verification
- `manage-labels`: Creates missing labels before the symptom references them
- `oc-auth`: Provides the Bearer token for the sippy-auth API
- `reevaluate-job-runs`: Optional dry-run testing and retroactive application
