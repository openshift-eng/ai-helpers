# PR Analysis Document Template

Use this template structure when producing the final PR-ANALYSIS.md. Adapt section depth and content to the actual PR — omit sections that have no data.

## Template

```markdown
# PR #<number>: <title>

> Auto-generated PR analysis. Language: <primary_language>. Modules affected: <count>. Files changed: <count>.

## Repository Overview

<2-3 paragraphs about what the repo does and its architecture. Sourced from an existing ONBOARDING.md or generated by the pr-repo-summarizer agent.>

## Pull Request Summary

<1-2 paragraphs synthesizing what this PR does, derived from the actual code change analyses. If the PR description or commit messages diverge from what the code actually does, note the discrepancy explicitly.>

| Field | Value |
|-------|-------|
| Author | <author> |
| Branch | <head> → <base> |
| Files changed | <count> |
| Modules affected | <count> |
| State | <open / merged / closed> |
| URL | <link> |

## Changes by Module

### <module-name>

<Detailed description of changes in this module: what changed, why, and how it affects the module's behavior. Synthesized from the pr-change-analyzer output.>

**Files changed:**
- `path/to/file` (+N, -M) — <summary of what changed>
- `path/to/other-file` (+N, -M) — <summary>

**Impact**: <How these changes affect the module's behavior>

**Risks**: <Potential issues or edge cases, if any>

## Cross-Module Impact

<How changes in one module affect or relate to changes in others. Identify shared dependencies, interface changes that ripple across modules, and coordination between module-level changes.>

## Risks and Considerations

<Aggregated risks from all module analyses. Include breaking changes, edge cases, error handling gaps, backwards compatibility concerns, and anything a reviewer should pay special attention to.>

## Files Outside Modules

<Changes to files not in any source module: configuration, CI/CD pipelines, documentation, build scripts, etc. Brief description of each.>

## Commit History

| SHA | Message |
|-----|---------|
| <short-sha> | <commit message> |
```

## Guidelines for the Synthesis Agent

- Write for an engineer reviewing this PR or joining the project mid-flight
- The Pull Request Summary should be readable standalone
- Trust the code change analyses over PR descriptions — the code is authoritative
- Use concrete function names and file paths, not abstract descriptions
- Risks should be specific and actionable
- Omit empty sections rather than writing "None" or "N/A"
- Keep the document under 2000 words for typical PRs, up to 3000 for large multi-module PRs
